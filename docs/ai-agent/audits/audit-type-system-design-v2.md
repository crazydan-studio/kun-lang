## 总体评价

Kun 的类型系统设计整体**一致且务实**。结构等价 + 无子类型 + HM 推断的组合是自洽的，避免了子类型与合一的根本性张力。效应跟踪通过 AST 标记而非类型级 IO monad 实现，降低了用户认知负担。等递归类型 + 别名机制提供了足够的表达能力。

但以下三个领域存在**设计空白**，可能在实现阶段引发问题：

## 发现清单

### 发现 #1：效应跟踪在高阶函数中的传播 [HIGH]

**参考文件**：`type-system.md:212-225`、`system-baseline.md:195-203`

**问题描述**：

效应跟踪方案定义了三类效应函数：(1) 含 `do` 块的函数、(2) 效应命名空间函数、(3) `Cmd.<bin>?` 等立即执行函数。纯函数不能调用效应函数。效应**不在类型签名中体现**。

但当效应函数作为回调传递给高阶函数时，传播语义不明确：

```kun
// Stream.iter 签名为纯函数类型
Stream.iter : (a -> Unit) -> Stream a -> Unit

// 但 IO.println 是效应函数，类型也是 String -> Unit
// 类型系统允许此调用，效应检查器需要额外处理
do
  files = Cmd.ls {} "."
  Stream.iter IO.println files
```

`Stream.iter` 接收一个类型为 `a -> Unit` 的函数参数。`IO.println : String -> Unit` 在类型上完全匹配。但 `IO.println` 是效应函数。效应检查器必须识别：传入效应回调给高阶函数后，该调用点变为效应性的。

**当前设计未定义**：
- 高阶标准库函数（`map`/`filter`/`iter`/`fold`）如何声明自己"传播效应参数"
- 用户定义的高阶函数如何处理效应回调
- 效应检查器是否需要类似 Koka 的"效应多态"（effect polymorphism）

**影响**：若仅通过 AST 扫描 `do` 块 + 效应命名空间调用来判定效应性，则 `Stream.iter IO.println files` 不会被标记为效应调用（`IO.println` 出现在参数位置而非调用位置），效应检查器可能漏报。未消费 Stream 等其他检查也有类似回调传播需求。

**建议**：
- 为高阶标准库函数标注"传播效应"属性（编译器内置知识）
- 或：用户定义的高阶函数若需接收效应回调，必须自身含 `do` 块并在 `do` 内调用回调
- 明确文档化此限制

### 发现 #2：C ABI 类型表示缺失关键布局定义 [HIGH]

**参考文件**：`system-baseline.md:329-395`

**问题描述**：

基础类型和 String/Bytes/List/Closure 的 C ABI 表示已定义。但以下类型的 C ABI 布局**未指定**：

| 缺失项 | 原因重要 |
|--------|---------|
| **Record 字段排序** | `{ name : String, age : Int }` 在内存中是声明顺序？字母序？对齐填充规则？影响 FFI 互操作 |
| **ADT tag 放置** | doc 说 `tag 使用 uint8_t`，但 tag 在 payload 之前还是之后？tag + payload 的对齐规则？变体大小是否取 max？ |
| **Tuple 布局** | `(Int, String, Bool)` 是否等同于匿名 Record？字段对齐规则？ |
| **`?T` 运行时表示** | Nil 如何编码？用 tagged union（`{ tag: u8, payload: union { void, T } }`）还是参考类型用 null 指针？ |
| **Map/Set 布局** | 仅描述为"哈希表"，无 C struct 定义 |

**影响**：FFI 调用 Kun 函数或 Kun 调用 C 函数时，结构体布局不一致将导致内存损坏。

**建议**：
- 定义 Record 字段按**声明顺序**排列，C 兼容对齐（alignof(max field)）
- ADT 采用 tag-first 布局：`{ uint8_t tag; union { ... } payload; }` 整体大小 = max variant size + tag + padding
- `?T` 对引用类型（String/List/等）使用 null 指针表示 Nil；对值类型使用 tagged union
- 补充 Map/Set 的 C ABI struct 定义

### 发现 #3：嵌套 Nilable 类型语义未定义 [MEDIUM]

**参考文件**：`type-system.md:49-84`

**问题描述**：

`Nil` 字面量的类型为 `?a`（多态）。当用于 `? ?Int` 类型时：

```kun
x : ? ?Int
x = Nil     // 这代表"外层为 Nil"还是"内层为 Nil"？

// 类型推断：Nil : ?a  合一  ? ?Int
// a = ?Int，所以 x : ? ?Int，表示外层为 Nil
```

由于 `Nil` 是 Nilable 类型的**单一值**，无法区分"外层存在但内层为 Nil"与"外层为 Nil"。这导致嵌套 Nilable 在语义上**折叠**：`? ?T` 的取值空间为 `{Nil, v | v : T}`，与 `?T` 的取值空间 `{Nil, v | v : T}` 完全相同。

这不是健全性 bug，但需要文档化：
- `? ?T` 是否合法？（当前文档未禁止）
- 若合法，其语义等同于 `?T`（值级别折叠）还是保留类型级别区分？
- `case x of Nil -> ...; Nil -> ...` 是否有二义性？

**建议**：文档中明确禁止嵌套 Nilable（`? ?T` 编译错误），或说明其折叠语义。推荐禁止，减少认知负担。

### 发现 #4：等递归类型深度限制的错误行为未定义 [MEDIUM]

**参考文件**：`type-system.md:258-260`

**问题描述**：

文档声明"编译器对递归 `type` 别名的展开有深度上限（默认 256 层），防止无限展开"。但未说明：
- 达到上限时编译器行为：**编译错误**（终止）还是**静默截断**（可能误接受错误程序）？
- 256 是递归别名展开深度还是类型结构比较深度？
- 交叉递归（A→B→A→B→...）展开计数规则：每个别名引用都计入？

**示例**：若为静默截断，以下程序可能被误接受：

```kun
type T = { x : ?T }
// 深度展开至 256 层截断，类型比较时可能将截断后的结构与目标匹配
```

**影响**：若采用静默截断，可能导致类型检查器接受实际类型不匹配的程序；若采用编译错误，256 深度对绝大多数场景足够。

**建议**：明确为**编译错误**（带信息："递归类型别名展开超过 256 层限制"），并说明跨别名引用均计入同一计数器。

### 发现 #5：穷举检查对 List rest-pattern 的覆盖性未指定 [MEDIUM]

**参考文件**：`type-system.md:19`、`syntax.md:466-480`

**问题描述**：

List 模式匹配支持 `[a, ..rest]`，穷举检查对 ADT 和 `Bool` 强制执行。但 List 的穷举规则未明确定义：

```kun
case list of
  [] -> 0                      // 长度=0
  [a, ..rest] -> 1             // 长度≥1
// 穷举：覆盖所有 List Int
```

问题：
1. 穷举检查器是否能识别 `[]` + `[a, ..rest]` 的**补集关系**？（这需要理解 `..rest` 匹配"剩余零或多个元素"的语义）
2. `[a, b]` + `[a, b, c, ..rest]` 是否被识别为覆盖长度 2 及 ≥3？（需要检查器理解前缀模式 + rest 的覆盖性）
3. **无限可能**：List 有无穷多种可能长度。若检查器仅做简单分支计数（如要求分支数 = 构造器数），则 List 永远无法通过穷举检查。检查器必须理解 `..rest` 作为"覆盖所有剩余"的语义。

**影响**：检查器算法若未专门处理 List rest-pattern 语义，会产生假阳性错误。

**建议**：文档化穷举规则：
- List 穷举要求"覆盖所有可能空/非空组合"，`[]` 或 `[..rest]` 作为通配基底
- `[a, ..rest]`（或仅 `[..rest]`）作为"非空所有长度"的覆盖
- 标准库 `List` 类型不应要求穷举（与 `Bool`/ADT 区分），或仅在存在穷举模式时要求

### 发现 #6：Let-多态与效应上下文的交互未定义 [MEDIUM]

**参考文件**：`system-baseline.md:160-168, 193-203`

**问题描述**：

HM let-多态在泛化（generalization）时将 let 绑定中的自由类型变量提升为多态。在纯函数式语言中这是健全的。但 Kun 有两个可能干扰的因素：

1. **效应函数**在 `do` 块内绑定值 → 若这些绑定被泛化（如通过 `let ... in`），效应可能被意外复制或隐藏
2. **延迟求值**（`let ... in` 绑定仅在引用时求值）→ 效应求值时机不确定

```kun
// 假设此代码合法（当前未明确是否禁止）
do
  let
    f = \x -> (do IO.println "side effect"; x)
  in
    (f 1, f 2)  // "side effect" 输出一次还是两次？
```

**关键问题**：`let ... in` 绑定是否允许包含效应代码？

当前文档暗示 `let ... in` 用于纯局部绑定（参考 Elm），`do` 块用 `=` 做顺序绑定。但并未明确禁止 `let ... in` 中出现效应函数调用。若允许，延迟求值 + 效应会产生与严格求值不同的语义，需要文档化。

**影响**：若 `let ... in` 允许效应代码 + 泛化 + 延迟求值，组合可能导致不可预测的行为。

**建议**：明确 `let ... in` 绑定的表达式必须是纯的（不能调用效应函数、不能包含 `do` 块）。使 `let ... in` 成为纯局部定义的专用结构，效应代码走 `do` 块。

### 发现 #7：效应函数 Lambda 的类型级不可见性 [LOW]

**参考文件**：`type-system.md:223`、`syntax.md:398`

**问题描述**：

"效应性不扩散到类型签名——函数签名中不出现 `IO` 标记"。这意味着：

```kun
// 这两个函数在类型上完全不可区分
pureAdd : Int -> Int -> Int
pureAdd = \x y -> x + y

effAdd : Int -> Int -> Int
effAdd = \x y ->
  do
    IO.println "adding"
    x + y
```

从类型签名的读者角度看，无法判断一个函数是否有副作用。这与 "显式错误" 和 "显式边界" 的设计原则存在张力——副作用恰恰是最需要显式化的属性之一。

**权衡分析**：设计文档已说明这是有意的取舍（避免 IO monad 污染类型签名，降低脚本场景的心智负担）。但代价是：
- 代码审查时需要检查函数体才能判断是否有副作用
- 模块公开 API 缺少效应文档（除非在文档注释中说明）
- IDE/工具无法仅凭类型签名提供效应提示

**建议**：考虑在文档生成（`kun doc`）中自动标注效应函数，弥补类型签名的不可见性。长期可评估是否为类型签名增加可选的效应标注语法。

### 发现 #8：结构等价下的类型抽象缺失 [LOW]

**参考文件**：`type-system.md:230-273`

**问题描述**：

结构等价意味着任何两个具有相同结构的类型都视为同一类型。`newtype` 提供了部分名义隔离，但仅适用于**单字段包装**。

无法创建"结构相同但语义不同"的多字段抽象类型：

```kun
// 模块 A
type User = { name : String, id : Int }

// 模块 B（独立开发，巧合相同结构）
type Session = { name : String, id : Int }

// 在模块 C：这两个类型是等价的！
// f : User -> String 可以接收 Session 值
```

在大型代码库中，独立模块可能定义结构巧合相同的类型，结构等价的全局性可能引发非预期的类型兼容。

**缓解因素**：
- `export` 控制字段可见性：若模块不导出字段名，外部无法构造该类型值，降低了意外混用的概率
- 需要语义隔离时用 `newtype` 包装整个 Record
- 脚本语言场景的模块图通常较小

**建议**：保持现状，文档中补充此权衡说明。

### 发现 #9：`?T` 字面量匹配的穷举规则未文档化 [LOW]

**参考文件**：`type-system.md:72-84`、`syntax.md:525`

**问题描述**：

文档声明穷举检查"对自定义和类型（含 `Result`）、`Bool` 强制穷举"。`?T` 具有类似和类型的结构（Nil 或 T），但其穷举规则未明确：

```kun
x : ?Bool
case x of
  Nil -> "none"
  true -> "yes"
  false -> "no"
// 这是穷举的吗？（覆盖了 Nil + Bool 的全部值？）

y : ?Int
case y of
  Nil -> "none"
  n -> toString n
// 这是穷举的吗？（Int 有无穷多个值，但 n 匹配所有）
```

对于 `?Int`，case 分支 `Nil` + `n` 显然是穷举的（n 覆盖所有 Int 值）。但对于 `?Bool`，写 `Nil` + `true` 是非穷举的（缺少 `false`）。

**问题**：穷举检查器对 Nilable 类型是否执行穷举检查？`?T` 是否像 ADT 一样被纳入强制穷举范围？`?Int` 的两个分支是否被视为穷举（因为 Int 有无限值且 n 捕获全部）？

**建议**：文档化：
- `?T` 类型的模式匹配**必须**包含 Nil 分支或通配符，否则编译警告/错误
- 非 Nil 分支的穷举性取决于 T 本身（Bool 需要 true+false；Int 用变量绑定即可）

### 发现 #10：零参纯函数退化与类型系统的一致性问题 [LOW]

**参考文件**：`type-system.md:176-201`、`syntax.md:393-398`

**问题描述**：

零参函数 `-> T` 仅允许用于效应函数。纯零参函数"退化为常量，应使用 `let` 绑定"。但类型构造器 `-> T` 本身不编码效应性，所以从类型系统的角度看：

```kun
// 合法：效应零参函数
now : -> DateTime
now = \ ->
  do
    Sys.time

// 若编译器允许，这也是合法的类型标注：
always42 : -> Int
always42 = \ -> 42  // 纯零参函数

// 但文档说"纯函数不允许定义为零参"
```

**问题**：类型检查器需要**额外规则**（非类型推导规则）来拒绝纯零参函数。这不是类型系统层面的限制，而是**语用限制**。若实现时忘记此规则，纯零参函数将通过类型检查。

**影响**：微小（仅涉及零参纯函数这一边界情况）。但在实现阶段需要注意。

**建议**：实现检查清单中标注此规则。

## 2026.06.14 结项附注
