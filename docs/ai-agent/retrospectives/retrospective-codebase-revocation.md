### 1. 撤销时未创建闭合审计/回顾/日志

- 决策过程无追溯——后人无法理解"为什么撤销"的具体动因
- 影响评估无记录——撤销波及哪些文档、是否还有残留引用未明确

**改进**：大规模撤销/重写决策必须配套创建闭合审计（验证撤销完整性 + 矛盾引用扫描）、回顾（沉淀经验教训）、日志（记录决策与执行）。

### 3. 撤销决策应在撤销前（而非事后）记录理由与影响评估

## 经验教训

### 1. 大规模撤销/重写决策必须有闭合文档

闭合文档三件套（audit + retrospective + log）不仅是任务完成的标志，更是决策可追溯性的关键。没有闭合文档的撤销会留下"知识黑洞"——后人无法理解为什么撤销、撤销是否完整、是否有残留影响。

### 2. 撤销后应立即审计所有文档的引用一致性

引用一致性扫描应作为撤销的标准操作流程的一部分。建议撤销后立即执行：
```
rg "code/kun-lang|src/.*\.zig" docs/ai-agent/
rg "已撤销|Phase [0-9]+ 完成|测试通过" docs/ai-agent/context/
```

### 3. 设计与实现分离维护时，设计大改动应触发实现撤销评估

本项目采用"设计先行 + 实现跟进"模式。当设计发生根本性改动（如废弃核心语法 `do`/`do in`、引入新类型系统 `effect`/`handler`）时，应主动评估实现是否需要撤销——而非继续在旧实现上打补丁。

### 4. "设计先行"原则：当设计发生根本性改动时，实现应等待设计稳定后再重写

本次撤销的根本原因是设计尚未稳定就推进了实现。如果设计阶段充分讨论并稳定后再实现，可避免大规模撤销的资源浪费。

### 5. 设计资产（文档）与代码资产（实现）应明确区分保留策略

撤销时，代码资产（`code/kun-lang/`）应当删除（避免误导），但设计资产（`docs/ai-agent/design/`、`architecture/`）应当保留（作为重新实现的依据）。本次撤销正确地区分了二者，是值得保留的经验。
