# 讨论记录：能力安全系统设计

## 讨论背景

Kun 最初的能力声明采用 `capability fs.read("/etc")` 形式，存在三个问题：语法上像函数调用但无执行语义、单复数不一致（`with capability` vs `with capabilities`）、三级粒度（脚本级/作用域块/单命令注解）过于复杂。需要重新设计能力系统的语法、语义和架构。

## 参与方

- AI Agent（方案分析与实施）
- 项目维护者（决策裁定）

## 讨论内容

### 议题 1：能力声明的语法形式

| 方案 | 要点 |
|------|------|
| `capability fs.read("/etc")`（旧） | 裸语句，伪函数调用，无执行语义 |
| `allow FsRead p"/etc"`（ADT 构造器） | 能力作为和类型，构造器大写，路径用 `p` 前缀 |
| `allow { fs = { read = [p"/etc"] } }`（Record 嵌套） | 三层嵌套对应三元组结构，但过于冗长 |
| `with caps fs.read = [p"/etc"]`（最终） | 扁平映射，`with caps` 关键字 + `命名空间.动作 = [目标列表]` |

**最终决策**：`with caps fs.read = [p"/etc"]`。理由：
- 直接映射运行时 `capability_check` 的三元组 `(Namespace, Action, Targets)`
- `with caps` 关键字独立成行，声明项缩进，格式统一
- 空列表 `[]` = 任意目标，替代有争议的 `Any` 关键字
- 无单复数区分（旧 `capability`/`capabilities` 问题）

### 议题 2：默认权限

| 方案 | 要点 |
|------|------|
| 默认读写工作目录（旧） | 便利但安全不严谨；写操作默认开放风险高 |
| 零默认能力 | 所有能力必须显式声明，包括当前目录读写 |

**最终决策**：**零默认能力**。脚本缺省没有任何能力。通过 `Path.cwd` 声明当前目录权限。

### 议题 3：声明层级

| 方案 | 要点 |
|------|------|
| 三级粒度（脚本级/作用域块/单命令注解） | 单命令注解语义上有歧义——`cmd with caps` 中的 `with` 作用于整个表达式还是仅命令本身？ |
| 二级粒度（脚本级/表达式级） | `with caps` 只附着于 `do`/`do in` 表达式，统一且无歧义 |

**最终决策**：**二级粒度**——仅脚本级和表达式级（`with caps ... do`）。理由：
- 单命令注解在语义上完全被作用域块覆盖（`with caps ... do` 包裹单条命令）
- 消除管道歧义（`a \|> b with caps` 中的 `with` 归属不明确）
- 作用域块只针对 IO 操作（`do` 块），与 IO 类型天然对齐

### 议题 4：能力对象的本质

| 方案 | 要点 |
|------|------|
| 标准库 ADT | `type Capability = FsRead [Path] \| FsWrite [Path] \| ...`，用户可见可操作 |
| 编译器内置结构 | `(Namespace, Action, Targets)` 三元组，编译器特殊解析 |

**最终决策**：**编译器内置结构**。`with caps` 块由编译器特殊解析，`fs.read = [...]` 仅在 `with caps` 块内有效。理由：
- 支持 `fs.read = [...]` 点号语法（非合法 identifier，无法用 Record 实现）
- 禁止动态构造和运行时修改——能力是编译期常量
- 编译期字段名校验（如 `fs.redaer` 拼写报错）

### 议题 5：能力与 OS 权限的关系

| 方面 | 决策 |
|------|------|
| 独立于 sudo | 能力系统与 OS sudo 正交，互不影响；不阻止脚本内部调用 sudo |
| 最低内核版本 | Linux 3.8（user namespace 支持非特权沙箱） |
| 不负责提权 | 能力系统不执行用户切换，由外部机制（sudo/polkit）处理 |

### 议题 6：单命令权限注解

| 立场 | 理由 |
|------|------|
| 保留 | 理论上提供最细粒度的控制 |
| 移除 | 语义上被 `with caps ... do` 完全覆盖；消除语法歧义；降低学习成本 |

**最终决策**：**移除**。无单命令权限注解。

### 议题 7：资源预算

| 方面 | 决策 |
|------|------|
| 归属 | 独立限流层，不属于能力系统 |
| 实现 | 执行器参数配置，缺省适配大多数脚本 |
| 控制对象 | CPU 时间、内存分配、执行时长 |

### 议题 8：CDF 能力声明

**最终决策**：CDF 不再声明能力行为。`behavior` 段从 CDF 格式中移除。seccomp 规则由参数类型和模式推导，而非能力声明。能力声明仅在可执行脚本和 `do` 表达式块中。

### 议题 9：审查机制

| 机制 | 说明 |
|------|------|
| `kun --audit` | 静态审计，展示脚本声明的所有能力而不执行 |
| `kun --confirm` | 交互式确认，逐项确认能力后执行 |
| `kun --cap-log` | 运行时审计，记录实际能力使用情况 |

**最终决策**：采用三种审查机制，覆盖审计、确认、追踪三个场景。

### 议题 10：目标字面量规则

| 动作 | 匹配规则 | 示例 |
|------|---------|------|
| `fs.read` | 路径前缀匹配 | `p"/etc/"` 匹配 `/etc` 及所有子路径 |
| `net.http` | 精确/glob 匹配 | `"*.example.com"` 匹配子域名 |
| `fs.meta` | （无目标） | `[]` 匹配任何文件 |

**最终决策**：目标只能是编译期字面量，不支持运行时动态拼接。

### 议题 11：模块能力规则

| 方面 | 决策 |
|------|------|
| 模块自身声明 | 禁止——模块中出现 `with caps` 是编译期错误 |
| 模块函数声明 | 允许——通过 `with caps ... do` 交集收窄 |
| 继承规则 | 模块函数的能力 = 调用方脚本能力 ∩ 函数自身 `with caps` |

**最终决策**：模块禁止声明能力，函数可收窄。

## 结论

1. 能力系统采用 `with caps` 语法，`命名空间.动作 = [目标列表]` 格式
2. 零默认能力——所有权限必须显式声明
3. 二级粒度——脚本级声明 + 表达式级 `with caps ... do`
4. 能力对象编译器内置，非标准库 ADT
5. 移除单命令权限注解
6. 移除 CDF 能力声明
7. 资源预算独立限流层
8. 三种审查机制（`--audit`、`--confirm`、`--cap-log`）
9. Linux 3.8 最低内核
10. 目标字面量编译期固定
