# 国际化（i18n）子系统

## 设计原则

1. **英文为首要语言**：msgid 为英文，英文 locale 下零额外存储开销（直接返回 msgid）
2. **中文内嵌**：zh_CN 翻译编译时嵌入二进制，无运行时文件依赖
3. **其他语言外挂**：非 zh_CN/en 的 locale 通过文件系统按需加载 `.po` 文件
4. **缺省一致**：任何 locale 下，未翻译的 msgid 返回英文原文，不 panic
5. **三路径格式化**：en/zh_CN 路径走 Rust proc macro + 运行时格式化（编译期校验占位符），external 路径走运行时字符串替换（构建时验证工具兜底校验）
6. **性能无感**：错误路径非热路径，翻译查找和格式化开销可忽略

## 消息 ID 体系

### msgid 格式

msgid 为英文文本，使用标准 gettext 约定：

- 简短标签：`"Type Mismatch"`、`"Expected"`
- 格式字符串：`"Type Mismatch: expected {s}, found {s}"`、`"at {f}"`
- 占位符使用 Rust `format!` 风格的位置参数：`{s}`（字符串）、`{d}`（整数）、`{f}`（源码位置）

### 消息分类

| 类别 | 示例 | 来源 |
|------|------|------|
| 编译期错误标题 | `"Type Mismatch"`、`"Effect Violation"` | 类型检查器 / 效应检查器 |
| 编译期错误字段标签 | `"Expected"`、`"Found"`、`"Reason"`、`"Hint"` | 结构化错误输出 |
| 编译期错误提示 | `"Did you forget a toString call?"` | 启发式建议 |
| 运行时 panic 消息 | `"interrupted by SIGINT"`、`"Recursion depth exceeded"` | 运行时 |
| CLI 输出 | `"Usage:"`、`"Options:"` | CLI 参数解析引擎 |

所有消息均以 `msgid` 标识，翻译存储在 `.po` 文件中。

## .po 文件

### 文件位置

```
kun-lang/
├── po/
│   ├── zh_CN.po       # 中文翻译（编译时嵌入）
│   └── README.md      # 翻译贡献指南
└── src/
    └── i18n/
        ├── catalog.rs   # 构建时生成：zh_CN 翻译表
        ├── locale.rs    # runtime locale 检测
        └── message.rs   # 消息格式化 API
```

### 格式约定

标准 GNU gettext `.po` 格式。每个条目包含：

```po
#: src/checker/type_error.rs:42
msgid "Type Mismatch"
msgstr "类型不匹配"

#: src/checker/type_error.rs:56
msgid "Expected"
msgstr "期望"

#: src/checker/type_error.rs:60
msgid "Type Mismatch: expected {s}, found {s}"
msgstr "类型不匹配：期望 {s}，实际为 {s}"
```

规则：

- `msgid` 精确匹配源码中的英文文本（含空白和占位符）
- `msgstr` 中占位符类型标识符（`s`/`d`/`f`）必须与 msgid 一致（数量和顺序必须相同——`format!` 按位置匹配）
- 内嵌翻译（zh_CN）的占位符在编译期由 Rust proc macro 校验；外挂翻译的占位符在构建时由 `cargo run --bin msgcheck` 验证（见下方构建时验证）
- 单条消息不超过一行；多行消息使用空行分隔
- 废弃条目（源码中已移除的 msgid）保留在 `.po` 文件底部，`#~` 标记

### 维护命令

```bash
# 从源码提取新增 msgid 到 .po
cargo run --bin msgmerge

# 检查 .po 翻译完成的条目
cargo run --bin msgcheck
```

## 构建时代码生成

### Cargo build.rs 集成

在 `build.rs` 中添加 `po2rs` 步骤：

```
build.rs 流程：
  src/**/*.rs ──→ parse source ──→ extract kmsg!("...", ...) calls
                                              │
  po/zh_CN.po ────────────────────────────────┤
                                              ▼
                                    generate src/i18n/catalog.rs
                                              │
                                              ▼
                                    正常 Rust 编译流程
```

`catalog.rs` 生成的内容：

```rust
// 自动生成，勿手动编辑
pub static TRANSLATIONS: &[(&str, &str)] = &[
    ("Type Mismatch", "类型不匹配"),
    ("Expected",       "期望"),
    ("Found",          "发现"),
    ("Reason: {s}",    "原因：{s}"),
    ("Hint: {s}",      "提示：{s}"),
    // ...
];
```

数组在编译期按 `msgid` 字典序排列，运行时通过二分查找定位。

### 性能特性

- 翻译表为编译期常量，分配在只读数据段（`.rodata`）
- 英文 locale 下直接返回 msgid，零查找开销
- 中文 locale 下二分查找，O(log n)，n 为消息总数（预估 < 200 条）
- 查找仅在错误/panic/CLI 输出时触发，非热路径

## 运行时 locale 检测

### 检测逻辑

```rust
pub enum Locale {
    En,
    ZhCN,
    External,  // 非 zh_CN/en 的其他 locale
}

pub fn detect_locale() -> Locale {
    // 1. KUN_LOCALE 显式覆盖（最高优先级）
    if let Some(val) = std::env::var_os("KUN_LOCALE") {
        if is_zh(&val.to_string_lossy()) { return Locale::ZhCN; }
        if is_en(&val.to_string_lossy()) { return Locale::En; }
        // 非 zh/en 的显式 locale：标记为 external
        return Locale::External;
    }
    // 2. POSIX locale 环境变量
    for var_name in ["LC_ALL", "LC_MESSAGES", "LANG"] {
        if let Some(val) = std::env::var_os(var_name) {
            let s = val.to_string_lossy();
            if is_zh(&s) { return Locale::ZhCN; }
            if is_en(&s) { return Locale::En; }
        }
    }
    // 3. 默认英文
    Locale::En
}

fn is_zh(s: &str) -> bool {
    s.starts_with("zh_CN") || s.starts_with("zh")
}

fn is_en(s: &str) -> bool {
    s.starts_with("en")
        || s.eq_ignore_ascii_case("C")
        || s.eq_ignore_ascii_case("POSIX")
}
```

### 检测结果的生命周期

`detect_locale()` 在运行时初始化阶段调用一次，结果存入 `RuntimeEnv`。整个进程生命周期内不变，无需加锁。

## 消息格式化 API

### 核心函数

消息格式化分为两层：

- **`kmsg`** — 纯模板查找（无分配），返回含 `{s}`/`{d}` 占位符的翻译字符串
- **`format`** — 模板查找 + 位置插值（分配最终字符串），统一处理三条 locale 路径

```rust
// message.rs

/// 返回指定 locale 下 msgid 的翻译字符串（含 {s}/{d} 占位符）。
/// en 路径直接返回 msgid——零分配、零查找。
pub fn kmsg(msgid: &'static str, locale: Locale) -> &'static str {
    match locale {
        Locale::En => msgid,
        Locale::ZhCN => lookup_zh_cn(msgid).unwrap_or(msgid),
        Locale::External => lookup_external(msgid).unwrap_or(msgid),
    }
}

/// 完成模板查找 → 位置插值，返回最终翻译字符串。
/// en/zh_CN 走 Rust format!（编译期校验占位符和参数类型）。
/// external 走运行时字符串替换（构建时 msgcheck 兜底校验占位符一致性）。
pub fn format(
    allocator: &mut bumpalo::Bump,
    locale: Locale,
    template: &'static str,
    args: &[FormatArg],
) -> Result<&str, FormatError> {
    match locale {
        Locale::En => {
            // 使用 format! 宏在编译期校验占位符与参数类型
            Ok(allocator.alloc_str(&format!("{}", /* template, args */)))
        }
        Locale::ZhCN => {
            let trans = lookup_zh_cn(template).unwrap_or(template);
            // zh_CN 翻译表在编译期校验，运行时同样使用 format!
            Ok(allocator.alloc_str(&format!("{}", /* trans, args */)))
        }
        Locale::External => {
            let trans = lookup_external(template).unwrap_or(template);
            runtime_replace(allocator, trans, args)
        }
    }
}
```

#### 为什么不能跳过 `format` 而用 `writeln!` 直接拼接？

Rust 的 `format!` 宏要求格式字符串和参数类型均在编译期确定：

```rust
// ✅ 编译期校验通过
writeln!(writer, "Type Mismatch: expected {}, found {}", expected, found)?;

// ❌ 编译错误：args 为运行时 slice，不是编译期已知的参数序列
let tmpl = kmsg("Type Mismatch: expected {s}, found {s}", locale);
writeln!(writer, tmpl, args_runtime)?;  // 错误：tmpl 不是字面量
```

错误消息中的参数值（如类型名、行号）是运行时构造的字符串，无法直接传给 `format!` 期望的编译期参数序列。必须经 `format` 先渲染为字符串再写入。此分配在 Arena 上完成，错误路径非热路径，开销可忽略。

### `format` 的使用示例

```rust
// 简单消息（无占位符）—— 仅需 kmsg
let title = i18n::kmsg("Type Mismatch", locale);

// 含位置插值的消息
let body = i18n::format(
    arena,
    locale,
    "Type Mismatch: expected {s}, found {s}",
    &[expected.into(), found.into(), m.span.into()],
)?;
```

### 运行时替换

external 路径的位置插值不使用 Rust `format!`，而是运行时字符串替换——将模板中的 `{s}`/`{d}` 替换为对应的位置参数：

```rust

    template: &str,
```

## 错误消息结构体

每个结构化错误在类型检查时构造，携带 msgid 和运行时参数，在输出阶段按 locale 格式化：

```rust
struct TypedErrorMessage {
    title_msgid: &'static str,  // 编译期常量，如 "Type Mismatch"
    location:    SourceLocation,  // file:line:col

    // 主体消息（可含 {s}/{d} 位置插值）
    body: MsgBody,

    // 结构化字段行（label: value 对）
    details: Vec<DetailLine>,
}

struct MsgBody {
    template: &'static str,  // msgid，如 "Type Mismatch: expected {s}, found {s}"
    args:     Vec<Arg>,       // 运行时参数
}

struct DetailLine {
    label_msgid: &'static str,  // "Expected"、"Found"
    value:       String,         // 运行时字符串值
}

struct Arg {
    // 位置参数，按模板中 {s}/{d} 出现顺序排列
    value: String,
}
```

`format_error` 将三部分按结构拼接输出：

```rust
fn format_error(err: &TypedErrorMessage, locale: Locale, writer: &mut impl Write) -> io::Result<()> {
    // 标题部分——无插值，直接 kmsg 后写入
    writeln!(writer, "{} ─── {}:{}:{}",
        i18n::kmsg(err.title_msgid, locale),
        err.location.file, err.location.line, err.location.col)?;

    // 主体部分——含位置插值，通过 format 渲染后写入
    let body = i18n::format(
        arena,
        locale,
        err.body.template,
        &err.body.args,
    )?;
    writeln!(writer, "{}", body)?;

    // 详细字段——固定格式，无插值
    for d in &err.details {
        writeln!(writer, "  {}: {}",
            i18n::kmsg(d.label_msgid, locale),
            d.value)?;
    }

    // 上下文代码行
    writeln!(writer, "  ──┤\n{}", err.location.context_line)?;
    Ok(())
}
```

输出示例（en locale）：

```
Type Mismatch ─── src/main.kun:12:20
  Expected: String, Found: Int
  ──┤
10 │   name : String
11 │   name =
12 │     42
    │     ^^
  ──┤
  Hint: Did you forget a toString call?  name = toString 42
```

此设计将"消息模板"与"运行时参数值"分离：
- `title_msgid`、`body.template`、`details[].label_msgid` 为编译期常量，`kmsg` 在此编译期完成翻译查找
- `body.args[].value` 和 `details[].value` 为运行时字符串，不做翻译
- `format_error` 对 `body` 走 `format`（因含位置插值，需要分配），对 `title` 和 `details` 走直接 `kmsg` + `writeln!`（无插值，无需分配）

### 各 locale 路径的处理流程

| 路径 | 模板来源 | 格式化方式 | 占位符校验时机 |
|------|---------|-----------|--------------|
| `.en` | `msgid` | `format!` | 编译期 |
| `.zh_CN` | 二进制内嵌翻译表 | `format!` | 编译期 |
| `.external` | 运行时加载 `.po` | 字符串替换 | 构建时 `msgcheck` |

## 构建时占位符验证

内嵌翻译（zh_CN）的占位符一致性由 Rust proc macro 在编译期校验——中文翻译中的 `{s}`/`{d}` 类型与 msgid 不一致时编译报错。

为 external 路径的外挂 `.po` 文件提供构建时验证工具 `cargo run --bin msgcheck`：

```bash
cargo run --bin msgcheck    # 验证 po/*.po 中所有翻译的占位符
```

验证规则：

| 规则 | 示例（非法） |
|------|-------------|
| `msgstr` 的占位符类型序列与 `msgid` 相同 | `msgstr "类型不匹配：期望 {d}，实际为 {d}"` → `{d}` 应与 msgid 的 `{s}` 一致 |
| `msgstr` 不可多出占位符 | `msgstr "期望 {s}，发现 {s}，位置 {s}"` → 多出 `{s}` |
| `msgstr` 不可缺少占位符 | `msgstr "期望 {s}"` → 缺少一个 `{s}` |

验证失败输出：

```
error: po/zh_CN.po:42 — placeholder mismatch for "Type Mismatch: expected {s}, found {s}"
  msgid placeholders:   {s}, {s}
  msgstr placeholders:  {s}, {s}, {s}
```

## 外挂 .po 加载

### 触发条件

仅当 locale 为 `external`（即 `KUN_LOCALE` 显式设置为非 zh/en 的值）时触发。环境变量自动检测不会返回 `external`。

### 加载流程

```
locale = "ja"
  │
  ├── 构造路径: <runtime>/share/kun/po/ja.po
  │
  ├── 安全检查:
  │     ├── 文件大小 < 1 MiB（拒绝异常大的 .po 文件）
  │     └── 路径无 ".."（防目录遍历）
  │
  ├── 解析 .po → 构建运行时哈希表
  │     ├── 解析成功 → 缓存到全局堆
  │     └── 解析失败 → stderr 警告 + fallback 英文
  │
  └── 后续 msgid 查找: 查哈希表 → 命中返回翻译; 未命中返回英文原文
```

### 安全边界

| 风险 | 防护 |
|------|------|
| 超大 .po 文件耗尽内存 | 文件大小上限 1 MiB |
| 路径遍历 (`../../etc/passwd`) | 拒绝含 `..` 的路径 |
| 格式错误的 .po 导致 panic | 解析为 best-effort：损坏条目跳过，不阻断进程 |
| 运行时覆盖二进制内嵌翻译 | 不检查：外挂仅作用于 external locale，zh_CN/en 始终使用内嵌 |

## 与其他子系统的集成

### 初始化阶段

locale 检测在运行时环境建立阶段执行（创建 Arena 后、模块解析前）：

```
运行时环境建立
  ├── 创建 Arena 分配器
  ├── 检测 locale（存入 RuntimeEnv）
  ├── 若 locale = external → 尝试加载外挂 .po
  ├── 设置全局求值环境
  └── 注册 Primitive 函数表
```

### 类型检查器集成

类型检查错误（`TypeError`）在构造时存储 msgid 和参数字段，不在构造时格式化。错误报告在输出阶段统一按 locale 格式化。

### CLI 集成

CLI 帮助文本使用 `kmsg` 输出。`--help` 输出自动跟随 locale。

### 标准库集成

标准库中暴露给用户的错误类型（`IOError`、`CommandError` 的 message 字段）为 Kun 字符串值，不做 i18n——由 Kun 用户代码自行处理。编译器和运行时的内部消息才纳入 i18n。
