# 国际化（i18n）子系统

## 设计原则

1. **英文为首要语言**：msgid 为英文，英文 locale 下零额外存储开销（直接返回 msgid）
2. **中文内嵌**：zh_CN 翻译编译时嵌入二进制，无运行时文件依赖
3. **其他语言外挂**：非 zh_CN/en 的 locale 通过文件系统按需加载 `.po` 文件
4. **缺省一致**：任何 locale 下，未翻译的 msgid 返回英文原文，不 panic
5. **三路径格式化**：en/zh_CN 路径走 Zig `std.fmt`（comptime 校验占位符），external 路径走运行时字符串替换（构建时验证工具兜底校验）
6. **性能无感**：错误路径非热路径，翻译查找和格式化开销可忽略

## 消息 ID 体系

### msgid 格式

msgid 为英文文本，使用标准 gettext 约定：

- 简短标签：`"Type Mismatch"`、`"Expected"`
- 格式字符串：`"Reason: {reason}"`、`"Hint: {hint}"`
- 占位符使用 `{name}` 风格，与 Zig `std.fmt` 的命名参数兼容

### 消息分类

| 类别 | 示例 | 来源 |
|------|------|------|
| 编译期错误标题 | `"Type Mismatch"`、`"Effect Violation"` | 类型检查器 / 效应检查器 |
| 编译期错误字段标签 | `"Expected"`、`"Found"`、`"Reason"`、`"Hint"` | 结构化错误输出 |
| 编译期错误提示 | `"Did you forget a toString call?"` | 启发式建议 |
| 运行时 panic 消息 | `"interrupted by SIGINT"`、`"Recursion depth exceeded"` | 运行时 |
| CLI 输出 | `"Usage:"`、`"Options:"` | CLI 参数解析引擎 |

所有消息均以 `msgid` 标识，翻译存储在 `.po` 文件中。

## .po 文件

### 文件位置

```
kun-lang/
├── po/
│   ├── zh_CN.po       # 中文翻译（编译时嵌入）
│   └── README.md      # 翻译贡献指南
└── src/
    └── i18n/
        ├── catalog.zig   # 构建时生成：zh_CN 翻译表
        ├── locale.zig    # runtime locale 检测
        └── message.zig   # 消息格式化 API
```

### 格式约定

标准 GNU gettext `.po` 格式。每个条目包含：

```po
#: src/checker/type_error.zig:42
msgid "Type Mismatch"
msgstr "类型不匹配"

#: src/checker/type_error.zig:56
msgid "Expected"
msgstr "期望"

#: src/checker/type_error.zig:60
msgid "Reason: {reason}"
msgstr "原因：{reason}"
```

规则：

- `msgid` 精确匹配源码中的英文文本（含空白和占位符）
- `msgstr` 中占位符 `{name}` 必须与 msgid 完全一致（数量、名称、顺序均可不同——命名插值不依赖顺序）
- 内嵌翻译（zh_CN）的占位符在编译期由 Zig `std.fmt` 校验；外挂翻译的占位符在构建时由 `zig build msgcheck` 验证（见下方构建时验证）
- 单条消息不超过一行；多行消息使用空行分隔
- 废弃条目（源码中已移除的 msgid）保留在 `.po` 文件底部，`#~` 标记

### 维护命令

```bash
# 从源码提取新增 msgid 到 .po
zig build msgmerge

# 检查 .po 翻译完成的条目
zig build msgcheck
```

## 构建时代码生成

### build.zig 集成

在 `build.zig` 中添加 `po2zig` 步骤：

```
build.zig 流程：
  src/**/*.zig ──→ parse source ──→ extract kmsg("...", ...) calls
                                              │
  po/zh_CN.po ────────────────────────────────┤
                                              ▼
                                    generate src/i18n/catalog.zig
                                              │
                                              ▼
                                    正常 Zig 编译流程
```

`catalog.zig` 生成的内容：

```zig
// 自动生成，勿手动编辑
pub const translations = [_]struct { msgid: []const u8, zh_CN: []const u8 }{
    .{ .msgid = "Type Mismatch", .zh_CN = "类型不匹配" },
    .{ .msgid = "Expected", .zh_CN = "期望" },
    .{ .msgid = "Found", .zh_CN = "发现" },
    .{ .msgid = "Reason: {reason}", .zh_CN = "原因：{reason}" },
    .{ .msgid = "Hint: {hint}", .zh_CN = "提示：{hint}" },
    // ...
};
```

数组在编译期按 `msgid` 字典序排列，运行时通过二分查找定位。

### 性能特性

- 翻译表为编译期常量，分配在只读数据段（`.rodata`）
- 英文 locale 下直接返回 msgid，零查找开销
- 中文 locale 下二分查找，O(log n)，n 为消息总数（预估 < 200 条）
- 查找仅在错误/panic/CLI 输出时触发，非热路径

## 运行时 locale 检测

### 检测逻辑

```zig
pub const Locale = enum {
    en,
    zh_CN,
    external,  // 非 zh_CN/en 的其他 locale
};

pub fn detectLocale() Locale {
    // 1. KUN_LOCALE 显式覆盖（最高优先级）
    if (std.os.getenv("KUN_LOCALE")) |val| {
        if (isZh(val)) return .zh_CN;
        if (isEn(val)) return .en;
        // 非 zh/en 的显式 locale：标记为 external
        return .external;
    }
    // 2. POSIX locale 环境变量
    for (&[_][]const u8{ "LC_ALL", "LC_MESSAGES", "LANG" }) |var_name| {
        if (std.os.getenv(var_name)) |val| {
            if (isZh(val)) return .zh_CN;
            if (isEn(val)) return .en;
        }
    }
    // 3. 默认英文
    return .en;
}

fn isZh(s: []const u8) bool {
    return std.mem.startsWith(u8, s, "zh_CN") or
           std.mem.startsWith(u8, s, "zh");
}

fn isEn(s: []const u8) bool {
    return std.mem.startsWith(u8, s, "en") or
           std.ascii.eqlIgnoreCase(s, "C") or
           std.ascii.eqlIgnoreCase(s, "POSIX");
}
```

### 检测结果的生命周期

`detectLocale()` 在运行时初始化阶段调用一次，结果存入 `RuntimeEnv`。整个进程生命周期内不变，无需加锁。

## 消息格式化 API

### 核心函数

消息格式化分为两层：

- **`kmsg`** — 纯模板查找（无分配），返回含 `{name}` 占位符的翻译字符串
- **`format`** — 模板查找 + 命名插值（分配最终字符串），统一处理三条 locale 路径

```zig
// message.zig

/// 返回指定 locale 下 msgid 的翻译字符串（含 {name} 占位符）。
/// en 路径直接返回 msgid——零分配、零查找。
pub fn kmsg(comptime msgid: []const u8, locale: Locale) []const u8 {
    return switch (locale) {
        .en => msgid,
        .zh_CN => lookupZhCn(msgid) orelse msgid,
        .external => lookupExternal(msgid) orelse msgid,
    };
}

/// 完成模板查找 → 命名插值，返回最终翻译字符串。
/// en/zh_CN 走 Zig std.fmt.allocPrint（编译期校验占位符和参数类型）。
/// external 走运行时字符串替换（构建时 msgcheck 兜底校验占位符一致性）。
pub fn format(
    allocator: Allocator,
    locale: Locale,
    comptime template: []const u8,
    args: anytype,
) ![]const u8 {
    return switch (locale) {
        .en => try std.fmt.allocPrint(allocator, template, args),

        .zh_CN => {
            const trans = comptime lookupZhCn(template) orelse template;
            try std.fmt.allocPrint(allocator, trans, args)
        },

        .external => {
            const trans = lookupExternal(template) orelse template;
            try runtimeReplace(allocator, trans, args)
        },
    };
}
```

#### 为什么不能跳过 `format` 而用 `writer.print` 直接拼接？

Zig 的 `std.fmt` 命名插值要求格式字符串和参数类型均在编译期确定：

```zig
// ✅ 编译期校验通过
try writer.print("Expected: {expected}, Found: {found}", .{
    .expected = "Int",
    .found   = "String",
});

// ❌ 编译错误：args 为运行时 slice，不是 comptime tuple
const tmpl = kmsg("Expected: {expected}, Found: {found}", locale);
try writer.print(tmpl, args_runtime);
```

错误消息中的参数值（如类型名、行号）是运行时构造的字符串 slice，无法直接传给 comptime 期望的 tuple。必须经 `format` 先渲染为字符串再写入。此分配在 Arena 上完成，错误路径非热路径，开销可忽略。

### `format` 的使用示例

```zig
// 简单消息（无占位符）—— 仅需 kmsg
const title = i18n.kmsg("Type Mismatch", locale);

// 含命名插值的消息
const body = try i18n.format(
    arena,
    locale,
    "Expected: {expected}, Found: {found}",
    .{ .expected = "Int", .found = "String" },
);
```

### 运行时替换

external 路径的命名插值不使用 Zig `std.fmt`，而是运行时字符串替换——将模板中的 `{key}` 替换为对应的值：

```zig
fn runtimeReplace(
    allocator: Allocator,
    template: []const u8,
    args: anytype,
) ![]const u8 {
    // args 为 struct { key: []const u8, value: []const u8 } 的切片
    // 对 template 做 {key} → value 的线性替换
    // 模板中不存在的占位符 → 保留原文，不 panic
    // args 中有但模板未使用的 → 忽略
}
```

## 错误消息结构体

每个结构化错误在类型检查时构造，携带 msgid 和运行时参数，在输出阶段按 locale 格式化：

```zig
const TypedErrorMessage = struct {
    title_msgid: []const u8,  // 编译期常量，如 "Type Mismatch"
    location:    SourceLocation,  // file:line:col

    // 主体消息（可含 {name} 命名插值）
    body: MsgBody,

    // 结构化字段行（label: value 对）
    details: []const DetailLine,
};

const MsgBody = struct {
    template: []const u8,  // msgid，如 "Expected: {expected}, Found: {found}"
    args:     []const Arg, // 运行时 KV 对
};

const DetailLine = struct {
    label_msgid: []const u8,  // "Expected"、"Found"
    value:       []const u8,  // 运行时字符串值
};

const Arg = struct {
    key:   []const u8,
    value: []const u8,
};
```

`formatError` 将三部分按结构拼接输出：

```zig
fn formatError(err: TypedErrorMessage, locale: Locale, writer: anytype) !void {
    // 标题部分——无插值，直接 kmsg 后写入
    try writer.print("{s} ─── {s}:{d}:{d}\n", .{
        i18n.kmsg(err.title_msgid, locale),
        err.location.file,
        err.location.line,
        err.location.col,
    });

    // 主体部分——含命名插值，通过 format 渲染后写入
    const body = try i18n.format(
        writer.context().allocator,
        locale,
        err.body.template,
        err.body.args,
    );
    try writer.print("{s}\n", .{body});

    // 详细字段——固定格式，无插值
    for (err.details) |d| {
        try writer.print("  {s}: {s}\n", .{
            i18n.kmsg(d.label_msgid, locale),
            d.value,
        });
    }

    // 上下文代码行
    try writer.print("  ──┤\n{s}\n", .{err.location.context_line});
}
```

输出示例（en locale）：

```
Type Mismatch ─── src/main.kun:12:20
  Expected: String, Found: Int
  ──┤
10 │   name : String
11 │   name =
12 │     42
    │     ^^
  ──┤
  Hint: Did you forget a toString call?  name = toString 42
```

此设计将"消息模板"与"运行时参数值"分离：
- `title_msgid`、`body.template`、`details[].label_msgid` 为编译期常量，`kmsg` 在此编译期完成翻译查找
- `body.args[].value` 和 `details[].value` 为运行时字符串，不做翻译
- `formatError` 对 `body` 走 `format`（因含命名插值，需要分配），对 `title` 和 `details` 走直接 `kmsg` + `writer.print`（无插值，无需分配）

### 各 locale 路径的处理流程

| 路径 | 模板来源 | 格式化方式 | 占位符校验时机 |
|------|---------|-----------|--------------|
| `.en` | `msgid` | `std.fmt` | 编译期 |
| `.zh_CN` | 二进制内嵌翻译表 | `std.fmt` | 编译期 |
| `.external` | 运行时加载 `.po` | 字符串替换 | 构建时 `msgcheck` |

## 构建时占位符验证

内嵌翻译（zh_CN）的占位符一致性由 Zig 编译器的 `std.fmt` 自动校验——中文翻译中的 `{name}` 若与 msgid 不一致，编译报错。

为 external 路径的外挂 `.po` 文件提供构建时验证工具 `zig build msgcheck`：

```bash
zig build msgcheck    # 验证 po/*.po 中所有翻译的占位符
```

验证规则：

| 规则 | 示例（非法） |
|------|-------------|
| `msgstr` 的占位符名称集合与 `msgid` 相同 | `msgstr "期望：{want}，得到 {got}"` → \`{want}\` \`{got}\` 与 msgid 的 \`{expected}\` \`{found}\` 不一致 |
| `msgstr` 不可多出占位符 | `msgstr "期望：{expected}，发现 {found}，位置 {pos}"` → \`{pos}\` 在 msgid 中不存在 |
| `msgstr` 不可缺少占位符 | `msgstr "期望：{expected}"` → \`{found}\` 缺失 |

验证失败输出：

```
error: po/zh_CN.po:42 — placeholder mismatch for "Expected: {expected}, Found: {found}"
  msgid placeholders:   {expected}, {found}
  msgstr placeholders:  {want}, {got}
```

## 外挂 .po 加载

### 触发条件

仅当 locale 为 `external`（即 `KUN_LOCALE` 显式设置为非 zh/en 的值）时触发。环境变量自动检测不会返回 `external`。

### 加载流程

```
locale = "ja"
  │
  ├── 构造路径: <runtime>/share/kun/po/ja.po
  │
  ├── 安全检查:
  │     ├── 文件大小 < 1 MiB（拒绝异常大的 .po 文件）
  │     └── 路径无 ".."（防目录遍历）
  │
  ├── 解析 .po → 构建运行时哈希表
  │     ├── 解析成功 → 缓存到全局堆
  │     └── 解析失败 → stderr 警告 + fallback 英文
  │
  └── 后续 msgid 查找: 查哈希表 → 命中返回翻译; 未命中返回英文原文
```

### 安全边界

| 风险 | 防护 |
|------|------|
| 超大 .po 文件耗尽内存 | 文件大小上限 1 MiB |
| 路径遍历 (`../../etc/passwd`) | 拒绝含 `..` 的路径 |
| 格式错误的 .po 导致 panic | 解析为 best-effort：损坏条目跳过，不阻断进程 |
| 运行时覆盖二进制内嵌翻译 | 不检查：外挂仅作用于 external locale，zh_CN/en 始终使用内嵌 |

## 与其他子系统的集成

### 初始化阶段

locale 检测在运行时环境建立阶段执行（创建 Arena 后、模块解析前）：

```
运行时环境建立
  ├── 创建 Arena 分配器
  ├── 检测 locale（存入 RuntimeEnv）
  ├── 若 locale = external → 尝试加载外挂 .po
  ├── 设置全局求值环境
  └── 注册 Primitive 函数表
```

### 类型检查器集成

类型检查错误（`TypeError`）在构造时存储 msgid 和参数字段，不在构造时格式化。错误报告在输出阶段统一按 locale 格式化。

### CLI 集成

CLI 帮助文本使用 `kmsg` 输出。`--help` 输出自动跟随 locale。

### 标准库集成

标准库中暴露给用户的错误类型（`IOError`、`CommandError` 的 message 字段）为 Kun 字符串值，不做 i18n——由 Kun 用户代码自行处理。编译器和运行时的内部消息才纳入 i18n。
