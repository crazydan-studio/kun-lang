# 需求综合指南

## 目的

将原始输入（用户反馈、问题报告、功能请求等）综合为结构化的需求文档。

## 需求文档结构

每个需求文档应包含以下部分：

### 1. 背景与动机

为什么需要这个功能？解决了什么问题？

### 2. 功能描述

详细描述功能的行为和预期效果。

### 3. 验收标准

明确的、可验证的验收条件。

### 4. 涉及模块

列出需要变更的模块和文件。

### 5. 约束与假设

技术约束、性能要求、假设条件。

### 6. 开放问题

尚未解决的问题和待决定的事项。

## 需求综合流程

1. 从 `docs/input/` 收集原始输入
2. 在 `docs/discussions/` 中澄清歧义
3. 编写结构化需求文档
4. 获取反馈并迭代
5. 确认需求基线

## 质量标准

- 需求必须可验证
- 需求之间不得矛盾
- 需求必须完整覆盖输入
- 验收标准必须明确
