# 计划编写与执行指南

## 何时需要编写计划

以下情况必须编写执行计划：
- 类型系统、命令签名、运行时等核心模块变更
- 涉及多个子系统
- 跨多个会话的长期任务
- 变更超过 5 个文件或约 200 行代码
- 需要分阶段执行
- 存在未解决的风险

## 计划结构

### 1. 背景与目标

简要描述为什么要做这个变更，期望达成什么目标。

### 2. 变更范围

明确列出：
- 涉及的模块
- 需要修改的文件
- 需要新建的文件
- 需要删除的文件

### 3. 实施步骤

按顺序列出具体的执行步骤，每个步骤应该：
- 足够具体，可直接执行
- 标注前置依赖
- 标注验证方法

### 4. 验证方法

列出验证变更正确性的方法：
- 单元测试
- 集成测试
- 手动验证
- 文档构建验证

### 5. 风险评估

识别可能的风险和缓解措施。

## 计划审计

所有计划在实施前必须经过独立审计。审计要点：
- 计划是否覆盖了需求的所有方面
- 实施步骤是否完整且有序
- 验证方法是否充分
- 风险是否被充分识别和缓解

## 计划执行

按照计划分步骤实施，每完成一个步骤：
1. 运行相关验证命令
2. 检查是否有意外影响
3. 更新计划状态
