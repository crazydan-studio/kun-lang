# 讨论记录：类型系统设计决策

## 讨论背景

类型系统核心设计需要确定多项关键决策，包括推断算法、整数语义、子类型策略、泛型约束、效应深度等，以为后续语法设计和实现提供基线。

## 参与方

- AI Agent（方案分析）
- 项目维护者（决策裁定）

## 讨论内容

### 议题 1：类型推断算法

| 方案 | 要点 |
|------|------|
| Hindley-Milner（算法 W） | 全程序推断，少标注；错误信息难读；Elm 已验证 |
| 双向类型检查 | 更优错误信息；需更多标注；实现复杂 |

**AI 观点**：推荐 HM，因为 Elm 是其语法主灵感源，HM+ADT+模式匹配已被大规模验证。

### 议题 2：整数语义

| 方案 | 要点 |
|------|------|
| 任意精度 | 无溢出；C ABI 不兼容；堆分配 |
| 固定宽度 i64 | C ABI 兼容；Zig 原生；溢出可运行时检查 |

**AI 观点**：推荐 i64，dlopen C ABI 为硬性约束。

### 议题 3：Nat 与 Int 的关系

| 方案 | 要点 |
|------|------|
| Nat <: Int | 便利；引入子类型化 |
| 独立类型，显式转换 | 类型系统纯净；便利性损失 |

**AI 观点**：推荐 Nat<:Int。**项目维护者裁定**：独立类型，无子类型。后续所有设计均服从此决策（无子类型为全局原则）。

### 议题 4：泛型约束

| 方案 | 要点 |
|------|------|
| 无约束（简单参数化多态） | 简单，推断稳定；无法表达 `show`/`sum` 等泛型操作 |
| 有约束（Typeclass/Trait） | 表达力强；实现复杂，MVP 范围外 |

**结论**：无约束。简单参数化多态，MVP 明确排除泛型约束。

### 议题 5：效应深度

| 方案 | 要点 |
|------|------|
| 仅标记 IO 边界 | 简单；能力安全系统已负责权限粒度 |
| 完整效应跟踪（Koka 风格） | 精确；与能力系统正交叠加 |

**结论**：仅标记 IO 边界。能力系统负责粒度，类型系统只回答"是否有副作用"。

### 议题 6：子类型

| 方案 | 要点 |
|------|------|
| 无子类型 | 纯净；与议题 3（Nat 独立）一致 |
| 有限子类型（仅 Nat<:Int） | 需特定场景支持 |

**结论**：无子类型。由议题 3 的 Nat 独立决策自然导出。

### 议题 7：Path 是否需要区分 File/Dir

| 方案 | 要点 |
|------|------|
| 单一 Path | 简单；路径操作本质是字符串操作 |
| 分离 File/Dir | 编译期安全；但路径类型在编译期常不可知 |

**结论**：单一 Path，以独立 `FileType` 枚举在运行时查询。

### 议题 8：正则修饰符语法

| 方案 | 要点 |
|------|------|
| 后缀标志（JS 风格） | 简洁；但需自定义解析 |
| 内联修饰符（PCRE 风格） | 标准；无语法扩展 |

**AI 观点**：推荐后缀标志。**项目维护者裁定**：仅支持内联修饰符。

## 结论

| 决策 | 最终选择 |
|------|---------|
| 类型推断 | Hindley-Milner（算法 W） |
| 整数语义 | 固定宽度 i64 |
| Nat/Int 关系 | 独立类型，显式转换 |
| 泛型约束 | 无约束（简单参数化多态） |
| 效应深度 | 仅标记 IO 边界 |
| 子类型 | 无 |
| Path/文件类型 | 单一 Path + FileType 运行时查询 |
| 正则修饰符 | 内联（PCRE 风格） |
