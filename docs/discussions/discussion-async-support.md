# 讨论记录：异步支持必要性分析

## 讨论背景

评估 Kun 是否需要内建异步支持（async/await、协程、Promise 等），以及如何在保持语言简洁的前提下处理并发脚本场景。

## 参与方

- AI Agent（方案分析）
- 项目维护者（决策裁定）

## 讨论内容

### 议题：是否需要在语言层面提供异步支持

#### 支持异步的理由

1. **批量操作加速**：同时 ping 多个主机、同时检查多个服务健康状态等场景，顺序执行延迟叠加
2. **事件等待**：监控脚本中同时等待多个文件变化、信号、定时器
3. **后台任务**：在继续主流程的同时处理耗时操作（如同时上传日志、清理临时文件）
4. **与同期语言对齐**：Python/Rust/JS/Go 均有内置异步或并发支持

#### 不支持异步的理由

1. **脚本场景 90%+ 是顺序的**：readFile → process → writeFile 天然顺序；命令管道（ls → grep → awk）也是顺序的
2. **引入复杂度**：事件循环、调度器、取消机制、死锁防护、错误传播、并发安全——这些对脚本语言是沉重负担
3. **Unix 哲学已有答案**：对于需要并发的场景，Shell 用户习惯使用 `&`/`wait`/`xargs -P`/`parallel`。Kun 也可以提供类似的库级解决方案而不侵入语言核心
4. **不需要新语法**：库级别的并发组合子（`parallel`、`race`、`timeout`）叠加在 `IO T` 上即可，不需要 `async`/`await` 关键字
5. **Stream 已解决部分问题**：惰性拉取的 Stream 天然支持反压，不需要 async 处理慢生产/快消费的问题
6. **最小维护面**：不引入 async 意味着不需要处理：线程池大小设置、取消传播、panic 跨任务传播、死锁检测、任务树管理

#### 方案对比

| 方案 | 复杂度 | 需要新语法 | 运行时变更 | 覆盖场景 |
|------|--------|-----------|-----------|---------|
| A: 标准库 `Task` 模块（组合子式并发） | 低 | 无 | 需要轻量线程或 poll 机制 | 批量执行、超时 |
| B: Structured Concurrency（`with task` 作用域） | 中 | 可能需 `with task` 关键字 | 需要任务树和取消传播 | 更安全的并发控制 |
| C: async/await 语法（全功能） | 高 | `async`/`await` 关键字 | 需要完整事件循环和执行器 | 全场景 |
| D: 不做（保持现状） | 无 | 无 | 无 | 脚本场景已足够 |

#### 方案 A 的候选 API

```
Task.parallel : List (IO a) -> IO (List a)
Task.race     : List (IO a) -> IO a
Task.timeout  : Duration -> IO a -> IO (Maybe a)
```

示例场景——同时检查多个服务：

```
checkServices = \addrs ->
  addrs
    // List (IO HealthStatus)
    |> map checkService
    // IO (List HealthStatus)
    |> Task.parallel
```

## 结论

**MVP 不做异步支持。** 理由：

1. 脚本语言的 90%+ 场景是顺序执行，不需要并发
2. 异步带来的语言复杂度（事件循环、调度器、取消、错误传播）远超其在脚本场景中的收益
3. 若未来需要，通过标准库 `Task` 模块（方案 A）添加组合子式并发，不需要语法变更

## 后续行动

- 无需记录到 backlog（不作为待办项跟踪）
- 若 MVP 后有明确需求，按方案 A（`Task` 模块）实现
- 决定可被未来的实际需求推翻——如果在脚本实践中发现并发是刚需，重新评估
