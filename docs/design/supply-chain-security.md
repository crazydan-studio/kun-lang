# 供应链安全

Kun 的命令签名系统和能力安全模型为防御供应链攻击提供了独特的基础设施。与传统的 Shell 脚本（命令以黑盒方式通过 fork/exec 执行）不同，Kun 在加载命令之前就通过 CDF（Command Description File）明确了命令的预期行为，这使得在命令执行前即可建立严格的验证和隔离机制。

## 攻击面分析

在 Kun 的架构下，供应链攻击可能针对以下环节：

| 攻击面 | 攻击方式 | 现有防御 |
|---|---|---|
| 命令二进制文件 | 恶意命令替换（如被篡改的 `curl`） | 二进制完整性校验 + CDF 签名预期 |
| CDF 文件 | 虚假的签名定义，掩盖命令的真实行为 | CDF 密码学签名 + 来源信任链 |
| 系统库（libc、openssl 等） | 恶意共享库注入 | 沙箱隔离 + seccomp 系统调用过滤 |
| 运行时自身 | Kun 解释器被篡改 | 解释器自身签名校验 |
| 项目级自定义签名 | 恶意 CDF 注入项目目录 | 项目签名信任策略 |
| 管道中间人 | 管道中被注入的命令窃取数据 | 管道类型约束 + 能力传递审计 |

## 防御方案

### 1. CDF 驱动的预期行为验证

CDF 定义了命令的预期输入、输出和错误类型。当命令的实际行为与 CDF 描述不一致时，运行时可以检测并拦截：

- **输出类型不匹配**：命令声明的输出类型为 `List<String>`（行列表），但实际返回了非文本二进制数据，运行时立即终止并报告 `OutputContractViolation` 异常
- **隐式文件访问**：CDF 中未声明文件访问依赖，但命令通过 seccomp 监控尝试了文件系统相关的系统调用，运行时记录审计日志并根据策略决定是否拦截
- **隐式网络连接**：CDF 中未声明网络依赖，但命令尝试建立 TCP 连接，运行时直接拦截并报告

CDF 本质上为每个命令建立了一份"行为契约"，运行时通过系统调用监控验证命令是否遵守了契约。任何超出契约的行为都是可疑信号。

### 2. 二进制完整性校验

在首次加载命令二进制时，运行时计算其密码学哈希（SHA-256）并记录。后续每次加载前重新校验，哈希不匹配时拒绝加载并报告 `BinaryIntegrityError`，附带：

- 预期哈希值
- 实际哈希值
- 命令路径
- 修改建议（检查是否为系统更新或恶意替换）

哈希数据库按层级维护：

```
~/.kun/trust/
├── system/           # 系统命令的已知良好哈希（由 Kun 发行版提供）
│   ├── ls.binhash
│   ├── cat.binhash
│   └── grep.binhash
├── project/          # 项目级自定义（提交到版本控制）
│   └── custom-tool.binhash
└── local/            # 用户本地信任（不提交）
    └── my-tool.binhash
```

### 3. Seccomp 系统调用过滤

基于 CDF 中的行为声明，运行时为每个命令动态生成 seccomp-BPF 过滤规则，将系统调用限制在 CDF 所描述的最小必要集合内：

| CDF 声明 | 允许的系统调用 | 禁止的系统调用 |
|---|---|---|
| 读取文件（`fs.read`） | `openat`、`read`、`fstat`、`close` | `connect`、`bind`、`execve` |
| 网络请求（`net.http`） | `socket`、`connect`、`send`、`recv` | `openat`（文件）、`execve` |
| 纯计算（无 IO 声明） | 仅内存操作 | 所有 IO 系统调用 |

这种从"声明到过滤规则"的自动推导机制，使得即便命令二进制被植入恶意代码（如后门、数据窃取），恶意行为也因系统调用被过滤而无法执行。CDF 越精确，seccomp 过滤就越严格，供应链攻击的成功窗口就越小。

### 4. CDF 密码学签名

CDF 文件支持密码学签名（如 Ed25519），确保签名定义的来源可信：

- **内置签名库**：由 Kun 发行版签名，使用发行版公钥验证
- **项目级 CDF**：由项目维护者签名，公钥存储在项目根目录的 `.kun/trusted-keys/` 中
- **社区贡献的 CDF**：通过信任链验证贡献者身份

验证流程：

```
加载 CDF → 验证签名 → 签名有效？→ 检查信任链 → 信任来源？→ 使用签名
               ↓                        ↓
          签名无效：拒绝加载          不可信来源：降级为自动推断 + 警告
```

### 5. 单命令沙箱隔离

对于高风险命令（如来自第三方安装、网络下载的命令），运行时可在独立的最小化 namespace 中执行该命令，实现命令级别的沙箱隔离：

- 独立的 Mount Namespace：仅挂载该命令所需的文件（基于 CDF 中的 `fs.read`/`fs.write` 声明）
- 独立的 PID Namespace：命令无法看到或操作其他进程
- 独立的 Network Namespace：仅开放 CDF 中声明的网络端点（基于 `net.http`/`net.tcp` 声明）

这与容器化的区别在于，隔离粒度是单个命令而非整个容器，启动开销极低（微秒级），且命令间的管道组合天然不受隔离影响（通过结构化数据传递而非文件描述符）。

### 6. 信任分级策略

Kun 对不同来源的命令采用分级信任策略，自动调整安全措施的严格程度：

| 信任级别 | 来源 | 安全措施 |
|---|---|---|
| `trusted` | 内置签名库 + 系统包管理器安装 | 类型检查 + 参数验证 |
| `verified` | CDF 密码学签名通过 + 二进制哈希匹配 | 类型检查 + 参数验证 + 输出契约验证 |
| `sandboxed` | 无签名或哈希不匹配 | 类型检查 + 参数验证 + seccomp 过滤 + 单命令沙箱 |
| `denied` | 已知恶意或签名密钥被吊销 | 拒绝加载 |

信任级别由运行时自动判定，也可由脚本通过权限声明显式覆盖（提升或降低）。

## 与传统方案的对比

| 维度 | 传统 Shell | Docker/容器 | Kun |
|---|---|---|---|
| 命令行为约束 | 无 | 无（容器级隔离） | CDF 行为契约 + seccomp |
| 二进制完整性 | 无 | 镜像哈希（容器级） | 单命令哈希校验 |
| 恶意命令检测 | 无 | 无 | 系统调用监控 + 契约违规检测 |
| 隔离粒度 | 无 | 整个容器 | 单个命令 |
| 启动开销 | 低 | 高（秒级） | 极低（微秒级） |
| 组合能力 | 管道（无类型） | 网络/卷挂载（重） | 类型化管道（轻量） |
